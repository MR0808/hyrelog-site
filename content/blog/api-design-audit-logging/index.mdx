# API Design for Audit Logging: Best Practices and Patterns

Integrating audit logging into your API design is crucial for security, compliance, and operational visibility. However, doing it well requires careful consideration of event structure, performance, reliability, and developer experience. Let's explore best practices for designing APIs that seamlessly incorporate audit logging.

## The Foundation: What to Log

Before designing your API, you need to understand what events matter for audit purposes. Generally, you should log:

### Authentication and Authorization Events

- Successful and failed login attempts
- Token issuance and revocation
- Password changes and resets
- Multi-factor authentication events
- Session creation and termination

### Data Access Events

- Reading sensitive data (customer information, financial records, PII)
- Creating new records
- Updating existing records
- Deleting records
- Exporting or downloading data

### Administrative Actions

- User management (creating, updating, deleting users)
- Permission and role changes
- System configuration changes
- Feature flag toggles
- Integration setup and changes

### Security Events

- Failed authorization attempts
- Rate limit violations
- Suspicious activity patterns
- API key creation and revocation

## Event Structure Design

A well-designed audit event has a consistent structure that makes it easy to query, analyze, and understand. Here's a recommended structure:

### Core Fields

Every audit event should include:

```json
{
  "actor": {
    "type": "user",
    "id": "user_123",
    "email": "alice@example.com",
    "ip_address": "192.168.1.100"
  },
  "action": "update",
  "resource": {
    "type": "customer",
    "id": "cust_456",
    "name": "Acme Corp"
  },
  "timestamp": "2025-02-01T10:30:00Z",
  "metadata": {
    "request_id": "req_789",
    "user_agent": "Mozilla/5.0...",
    "method": "PATCH",
    "path": "/api/v1/customers/cust_456"
  }
}
```

### Actor Identification

The actor field identifies who performed the action. It should include:
- **Type**: What kind of actor (user, service, system, api_key)
- **ID**: A unique identifier
- **Additional context**: Email, username, or other identifying information
- **Source**: IP address, device information, location (when available)

### Action Verbs

Use consistent, clear action verbs:
- `create`, `read`, `update`, `delete` for CRUD operations
- `login`, `logout`, `authenticate` for auth events
- `export`, `import`, `download` for data movement
- `grant`, `revoke`, `modify` for permission changes

Avoid ambiguous verbs like `do`, `perform`, or `execute`. Be specific about what happened.

### Resource Context

Include enough context about the resource to make the event meaningful:
- **Type**: What kind of resource (customer, order, configuration)
- **ID**: Unique identifier
- **Name or description**: Human-readable identifier
- **Before/after states**: For update events, include what changed

### Metadata for Debugging

Include request-level metadata that helps with debugging and investigation:
- Request ID for tracing
- User agent
- HTTP method and path
- Response status code
- Error messages (for failures)

## API Integration Patterns

There are several patterns for integrating audit logging into your API:

### Pattern 1: Synchronous Logging

Log events synchronously as part of the request handling:

```typescript
async function updateCustomer(customerId: string, data: CustomerUpdate) {
  const customer = await getCustomer(customerId);
  
  // Perform the update
  const updated = await db.customers.update(customerId, data);
  
  // Log the event synchronously
  await auditLog.log({
    actor: getCurrentActor(),
    action: 'update',
    resource: {
      type: 'customer',
      id: customerId,
      name: customer.name
    },
    changes: computeChanges(customer, updated)
  });
  
  return updated;
}
```

**Pros**: Simple, guarantees logging happens
**Cons**: Adds latency to requests, can slow down API responses

### Pattern 2: Asynchronous Logging

Log events asynchronously to avoid impacting request performance:

```typescript
async function updateCustomer(customerId: string, data: CustomerUpdate) {
  const customer = await getCustomer(customerId);
  const updated = await db.customers.update(customerId, data);
  
  // Log asynchronously (fire and forget)
  auditLog.log({
    actor: getCurrentActor(),
    action: 'update',
    resource: { type: 'customer', id: customerId, name: customer.name },
    changes: computeChanges(customer, updated)
  }).catch(err => {
    // Log error but don't fail the request
    logger.error('Failed to log audit event', err);
  });
  
  return updated;
}
```

**Pros**: Doesn't impact request latency
**Cons**: Risk of losing events if the system crashes, more complex error handling

### Pattern 3: Event Sourcing

Use event sourcing where audit events are the source of truth:

```typescript
async function updateCustomer(customerId: string, data: CustomerUpdate) {
  const customer = await getCustomer(customerId);
  const changes = computeChanges(customer, data);
  
  // Emit event first
  const event = await eventStore.append({
    type: 'customer.updated',
    actor: getCurrentActor(),
    resource: { type: 'customer', id: customerId },
    changes: changes,
    timestamp: new Date()
  });
  
  // Update database based on event
  await applyEventToDatabase(event);
  
  return getCustomer(customerId);
}
```

**Pros**: Complete audit trail, can replay events, very powerful
**Cons**: Significant architectural change, more complex to implement

### Pattern 4: Middleware-Based Logging

Use middleware to automatically log API requests:

```typescript
app.use(auditLoggingMiddleware({
  includeBody: false, // Don't log request bodies by default
  includeResponse: false,
  filter: (req) => {
    // Only log certain endpoints
    return req.path.startsWith('/api/v1/customers') ||
           req.path.startsWith('/api/v1/orders');
  }
}));
```

**Pros**: Centralized, consistent, easy to add/remove
**Cons**: Less control over event structure, may log too much or too little

## Performance Considerations

Audit logging can impact API performance if not designed carefully:

### Minimize Synchronous Work

Avoid blocking requests on audit logging. Use asynchronous patterns or background workers when possible.

### Batch Events

When logging many events, batch them together:

```typescript
// Instead of logging each item individually
for (const item of items) {
  await auditLog.log({ ... });
}

// Batch them
await auditLog.logBatch(
  items.map(item => ({ ... }))
);
```

### Use Efficient Storage

Choose storage systems optimized for write-heavy workloads. Consider time-series databases or specialized audit log services.

### Sample When Appropriate

For very high-volume, low-value events, consider sampling:

```typescript
if (shouldSample(event)) {
  await auditLog.log(event);
}
```

## Reliability and Durability

Audit logs must be reliable—lost events can't be recovered. Consider:

### Idempotency

Make audit logging idempotent so retries don't create duplicate events:

```typescript
await auditLog.log({
  idempotency_key: `${requestId}-${timestamp}`,
  actor: getCurrentActor(),
  action: 'update',
  resource: { ... }
});
```

### Acknowledgment Patterns

Use acknowledgment patterns to ensure events are persisted:

```typescript
const eventId = await auditLog.log(event);
await auditLog.awaitAcknowledgment(eventId, { timeout: 5000 });
```

### Retry Logic

Implement retry logic for failed log writes:

```typescript
async function logWithRetry(event: AuditEvent, maxRetries = 3) {
  for (let i = 0; i < maxRetries; i++) {
    try {
      await auditLog.log(event);
      return;
    } catch (err) {
      if (i === maxRetries - 1) throw err;
      await sleep(1000 * (i + 1)); // Exponential backoff
    }
  }
}
```

## Security Considerations

Audit logs themselves contain sensitive information and must be protected:

### Don't Log Secrets

Never log passwords, API keys, tokens, or other secrets:

```typescript
// BAD
await auditLog.log({
  action: 'login',
  password: userPassword // NEVER DO THIS
});

// GOOD
await auditLog.log({
  action: 'login',
  actor: { email: userEmail },
  success: true
});
```

### Sanitize Sensitive Data

Redact or hash sensitive data in logs:

```typescript
await auditLog.log({
  action: 'update',
  resource: {
    type: 'customer',
    id: customerId,
    // Don't log full credit card numbers
    credit_card: maskCreditCard(customer.creditCard)
  }
});
```

### Control Access to Logs

Limit who can read audit logs. They should be more restricted than regular application logs.

### Encrypt at Rest

Encrypt audit logs when stored, especially if they contain sensitive information.

## Testing Audit Logging

Test your audit logging just like you test other functionality:

### Unit Tests

Test that events are logged with correct structure:

```typescript
test('logs customer update event', async () => {
  const logSpy = jest.spyOn(auditLog, 'log');
  
  await updateCustomer('cust_123', { name: 'New Name' });
  
  expect(logSpy).toHaveBeenCalledWith(
    expect.objectContaining({
      action: 'update',
      resource: { type: 'customer', id: 'cust_123' }
    })
  );
});
```

### Integration Tests

Verify events are actually persisted and queryable:

```typescript
test('audit event is queryable after creation', async () => {
  await updateCustomer('cust_123', { name: 'New Name' });
  
  const events = await auditLog.query({
    resource: { type: 'customer', id: 'cust_123' },
    action: 'update'
  });
  
  expect(events).toHaveLength(1);
  expect(events[0].resource.id).toBe('cust_123');
});
```

## Common Mistakes to Avoid

### Logging Too Much

Don't log every single API call—focus on business-significant events. Logging every read operation can create noise and performance issues.

### Logging Too Little

Don't skip important events. If you're not sure whether to log something, err on the side of logging it.

### Inconsistent Structure

Use a consistent event structure across your entire API. Inconsistency makes querying and analysis difficult.

### Ignoring Failures

Don't silently fail audit logging. At minimum, log failures to your application logs so you know when audit logging isn't working.

### Not Testing

Test your audit logging. Broken audit logging can be worse than no audit logging if it gives false confidence.

## Conclusion

Well-designed audit logging is essential for modern APIs. By following these best practices—consistent event structure, appropriate integration patterns, performance considerations, and security measures—you can build audit logging that supports security, compliance, and operations without compromising API performance or developer experience.

The key is to think about audit logging from the start of your API design, not as an afterthought. When integrated properly, audit logging becomes a natural part of your API that provides immense value for security, compliance, debugging, and understanding how your system is actually being used.

Whether you're building audit logging into your existing API or designing a new one, these patterns and practices will help you create a robust, reliable audit trail system that serves your organization's needs.
